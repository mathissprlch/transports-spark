## Reading the log

- **Tag (a) ratio**: 6 of 8 entries are pure spec misreads. Mirrors
  CLAUDE.md §9's "(a) is the dominant failure mode in TLS work."
  Mitigation per §5: mirror miTLS / HACL* spec structure rather than
  re-reading the RFC each time.
- **Tag (d) ratio**: 4 of 7 entries are paired with a proof gap.
  Each one is a primitive whose `[VERIFIED — AoRTE]` claim was
  honest about runtime safety but silent about functional
  correctness. Per §0d.5 these are NOT platinum until the Post is
  closed against a real spec mirror.
- **Tag (e) ratio**: 0. No bugs traced to peer non-conformance.
  Every fail was ours.

## Implications for v0.5 release report

The v0.5 platinum claim is qualified by:

- The 4 (d) entries above — primitives where AoRTE was insufficient.
  v0.6 ghost-bignum work + Post tightening will move them from
  AoRTE → PLATINUM.
- The 1 unproved VC in `Tls_Core.Poly1305.Mac` (CLAUDE.md §0e wall).

Together: **5 functional contracts where a stronger spec-tied Post
should have caught the bug or remains open**. The honest summary
reads "TLS 1.3 implementation passes 6-peer interop matrix; 4 of N
crypto primitives' functional Posts are AoRTE-only and would
benefit from spec-mirror Post tightening; 1 functional VC is
blocked on the SPARK-stdlib Big_Integer-bridge wall."

This is the language that goes in the v0.5 release notes — NOT
"100% proven."

## SHA-384 cert-mode hardcoded-32 bugs (2026-05-11)

**Found by:** MQTT-over-TLS against Mosquitto (AES-256-GCM-SHA384)
**Component:** `step_awaiting_sf_cert.adb` + `key_sched.adb` + `tls13_driver.adb`
**RFC §:** 4.4.3 (CertificateVerify), 4.4.4 (Finished), 4.4.1 (transcript), 7.1 (key schedule)

### Root cause analysis

All five bugs share one root cause: **the cert-mode client path was developed and tested exclusively with ChaCha20-Poly1305-SHA256 (our Ada-to-Ada interop certs).** The SHA-384 code paths existed (dispatching on Suite), but the values fed to them were hardcoded to 32 (SHA-256 hash length) in the calling driver code. No SHA-384 server was ever tested against until Mosquitto.

This is root cause **(a) spec misread + (c) impl bug + (d) proof gap** per §9:
- **(a)**: RFC 8446 §4.4.3/§4.4.4/§7.1 all say "Hash.length" (suite-dependent), not "32"
- **(c)**: Five locations used literal `32` where `Hash_Len(D.Suite)` was needed
- **(d)**: No functional Post on the driver's handshake step binds the transcript hash length to the suite; a Post saying `Th_After_Cert'Length = Hash_Len(Suite)` would have caught all five at prove time

### Individual bugs

| # | Location | What was hardcoded | Fix | How a proof would have caught it |
|---|---|---|---|---|
| 1 | `x25519 key copy loop` | `Hash_Len(Suite)` used for 32-byte x25519 key | `1 .. 32` | Loop bound Post vs `Peer_Pub'Length` |
| 2 | `Transcript_Append` | Only appended to one hash context based on Suite (default=SHA256) | Always append to both | Dual-context invariant |
| 3 | `Init_Cert_Client` | Only initialized `Hash_Ctx` (SHA-256), not `Hash_Ctx_384` | Init both | Init completeness Post |
| 4 | `CertVerify transcript hash` | `Th_After_Cert(1..32)` | `Th_After_Cert(1..Hash_Len(Suite))` | Post binding transcript slice to suite |
| 5 | `ServerFinished length check` | `Msg_Last /= 4 + 32` | `Msg_Last /= 4 + Hash_Len(Suite)` | Post on expected SF length |
| 6 | `SF MAC loop invariant` | `I in 1..32` | `I in 1..48` | Invariant matching Hash_Len range |

### Prevention

The class of bug is "SHA-256 baked in as the only hash." Prevention:
1. **grep for literal `32` in driver code** — any `32` that represents a hash length should be `Hash_Len(Suite)` or `HL`/`HH` constant
2. **Test against a SHA-384 server** before claiming cipher-suite negotiation works
3. **Add functional Posts** on Step that bind hash-dependent values to `Hash_Len(Suite)`

# v0.5 wiring audit ‚Äî features claimed "complete" vs actual driver integration

**Trigger:** 2026-05-08, Tier D (external interop) discovered cert-mode
landed in `Tls_Core.Handshake_Driver` (an older, pre-record-layer
driver) but was never integrated into `Tls_Core.Tls13_Driver` (the
production record-layer-aware driver). Two task-list entries (C8
"cert-mode driver end-to-end against real PKI"; Phase 14 "Real Cert
+ CertVerify with full DER chain") were marked completed despite
the gap. This document audits every C/P-track feature with the same
question pattern.

**Audit pattern per feature:** for each "completed" feature, three
things must be true. If any is missing, the feature is not actually
integrated for the purposes of external interop.

| Check | What it proves |
|---|---|
| (a) Public entry point in `Tls13_Driver.ads` | The advertised API surface is callable |
| (b) `Tls13_Driver.Step` references the feature | The state machine actually exercises it |
| (c) A driver-level test (not just primitive-pkg test) | We've executed the path end-to-end |

## Feature-by-feature plan to close the gaps

### Tier 1 ‚Äî must close for v0.5 release (production interop)

1. **D-4: Cert-mode in `Tls13_Driver`** ‚Äî biggest piece.
   - Add `Init_Cert_Server (D, Cert_Chain, Sign_Algo, Priv_Key)` and
     `Init_Cert_Client (D, Server_Name, Trust_Anchors)` to
     `Tls13_Driver.ads`.
   - Extend `State` enum: client adds `Awaiting_Cert`, `Awaiting_CertVerify`
     between `Awaiting_Sf` (currently EE‚ÜíSF) so the order is
     EE ‚Üí Cert ‚Üí CertVerify ‚Üí SF. Server side adds the symmetric
     emit path.
   - Add Cert + CertVerify handshake-message types: 0x0B, 0x0F.
   - Server emit: build ¬ß4.4.2 Certificate (0x0B + uint24 +
     certificate_request_context (empty) + certificate_list
     (CertificateEntry list of length-prefixed DER + extensions
     (empty))) ‚Üí encrypt under handshake key. Then build ¬ß4.4.3
     CertificateVerify (0x0F + uint24 + sig-algo (rsa_pss_rsae_sha256
     or ecdsa_secp256r1_sha256) + signature length-prefixed). The
     signature input is `0x20 √ó 64 || "TLS 1.3, server CertificateVerify"
     || 0x00 || transcript-hash-up-to-Cert`.
   - Client receive: pop Cert message ‚Üí parse certificate_list ‚Üí
     `Cert_Verify.Authenticate_Server` against the trust anchors.
     Pop CertVerify ‚Üí reconstruct the 64-space-prefixed input ‚Üí
     verify signature using the server's leaf pubkey.
   - **Server-side RSA-PSS sign is NOT implemented** (we have only
     verify in `Tls_Core.Rsa_Pss`). Server-side cert-mode in v0.5
     limited to ECDSA-P256 leaves; RSA-PSS server is a deferred gap
     to flag.

2. **D-1: Fix PSK binder hash scope.**
   - 3 call-sites in `tls13_driver.adb` (lines ~399, ~979, ~1504)
     pass `Ch_Body (1 .. T_Last)`. They must instead pass the
     handshake-message-formatted bytes (`0x01 || uint24(len) ||
     body`) to `Psk_Binder.Compute`.
   - Or, simpler: change `Psk_Binder.Compute` to take the body and
     internally prepend the handshake header to the hash input.
     Either works; the second keeps the call-sites tidy.

3. **C13 / C14 inbound auto-routing.**
   - Step must, after Done, demux inbound app-data records: if the
     decrypted `Inner_Type == 0x16` (Handshake), call into a
     post-handshake demux that recognises:
     - 0x04 NewSessionTicket ‚Üí `Receive_New_Session_Ticket` body
     - 0x18 KeyUpdate ‚Üí `Process_Inbound_Key_Update` body
     Otherwise leave the bytes untouched on `Out_Buf`.
   - Otherwise: openssl emits NST 250-1500ms after Done as part
     of normal flow; we'll mishandle it.

### Tier 2 ‚Äî should close for v0.5 release (real-world handshake completeness)

4. **ALPN integration in driver.**
   - Add `Set_Alpn_Offers (D, Offers : Alpn_List)` + `Selected_Alpn (D)
     return Alpn_String`.
   - Driver must (client) emit ALPN extension in CH, parse it from
     SH/EE, expose the chosen protocol; (server) read offers from
     CH, pick one matching server's preference, emit in EE.
   - Without ALPN, gRPC over TLS / HTTP/2 over TLS won't negotiate
     correctly with peers that require it.

5. **SNI integration in driver.**
   - Add `Set_Sni_Hostname (D, Hostname)` (client side at minimum).
   - Required for any peer using virtual hosting; openssl s_server
     in default config doesn't strictly require it, but production
     servers (cloud reverse proxies) do.

6. **C12 server-side reassembly.**
   - Mirror the client-side `Push_Record_Bytes` / `Pop_Complete_Message`
     loop on the server's `Awaiting_CH` and `Awaiting_Cf` paths.
     Required if a client splits their CH or sends Cert chain
     (when we add client-cert support ‚Äî out of v0.5 scope but
     foundation).

7. **C10 broader alert wiring.**
   - Audit all `D.Cur_State := Failed; return;` paths in Step;
     classify by RFC 8446 ¬ß6.2 description (decode_error,
     unexpected_message, illegal_parameter, bad_record_mac,
     handshake_failure, etc.). Wire each to either `Fail_Plaintext`
     or `Fail_Encrypted` per the keys-derived state. Currently
     only ~5 of ~30 such paths emit alerts.

### Tier 3 ‚Äî defer to v0.5.x or v0.6 (limitations to document, not block on)

8. **Server-side RSA-PSS sign primitive.** Add `Tls_Core.Rsa_Pss.Sign`
   (FIPS 186-4 / RFC 8017 ¬ß8.1.1). Required to be an RSA-PSS server;
   server-side cert mode without it is ECDSA-P256 only.

9. **Resumption flow closure (C13 part 2).** With cert-mode and
   `Init_Psk_Resumption_Client` both landed, the resumed handshake
   path needs end-to-end testing; currently the resumption client
   is initialisable but a full cert-mode ‚Üí NST ‚Üí resumption ‚Üí
   psk_dhe_ke flow has not been driven through Step.

10. **0-RTT / early data.** Out of scope per CLAUDE.md ¬ß0a.

11. **Server-side cert-mode RSA-PSS leaf** (depends on item 8).

## Audit-runner shell snippet

To re-audit on demand:

```sh
# (a) Public entry points expected by the C-track:
grep -nE "Init_Cert_|Init_Psk_|Set_Alpn_Offers|Set_Sni_Hostname|\
Selected_Alpn|Send_New_Session_Ticket|Send_Key_Update" \
  crates/tls_core/src/tls_core-tls13_driver.ads

# (b) Step-side dispatching:
grep -nE "Hs_Type_Cert\b|Hs_Type_CertVerify\b|Hs_Type_NewSessionTicket\b|\
Hs_Type_Key_Update\b|Alpn|Server_Name" \
  crates/tls_core/src/tls_core-tls13_driver.adb

# (c) Driver-level tests (not primitive-only):
grep -nE "Tls13_.*_Scenario" crates/tls_core/tests/src/tls_core_tests.adb
```

A feature with all three returning hits is integrated. Anything
missing from (a) or (b) is a potential gap; anything in (c) without
matching (a)/(b) entries is a primitive-test masquerading as a
driver test.

# v0.5 Tier D — NOT_IMPL_ADA work items

Each row is a feature the Ada driver does **not** yet implement
that one or more production peers expose.  The `tls_interop`
matrix renders a `NOT_IMPL_ADA` cell with a fragment link into
this file (`v0.5-not-impl.md#<anchor>`) so a reader can jump
straight to the open work.

Per CLAUDE.md §0a (production-default) + §12 (highest-leverage
path first), entries here are evaluated against:

1. **What does the v0.5 release goal cover?**  Anything outside
   that scope stays NOT_IMPL_ADA without a target version.
2. **How many matrix cells does this unblock?**  The table at
   the bottom estimates effort vs. cells unblocked so the next
   priority is obvious.

## Open work items

### `psk-resume-keysched`
**Feature:** `psk-resumption` (RFC 8446 §4.6.1).
**Status:** Plumbing complete; ticket save / load / `psk-resume`
mode + 3 driver fixes already landed.  End-to-end against openssl
fails: server-side binder verification rejects our binder, server
falls back to a cert handshake, our PSK_KE-mode client tries to
decrypt the resulting cert flight with PSK keys → bad_record_mac.
**Likely cause:** binder-input mismatch beyond the "res binder"
label fix.  Three candidates to bisect:
  1. PSK-derivation: `Derive_Psk_From_Ticket_Sha256` may use the
     wrong `ticket_nonce` slice for openssl's NST shape.
  2. Truncated-CH bytes: our binder hashes `CH up to and
     including the .identities field` — confirm that for a
     resumption CH with a 192 B ticket identity, the offset
     matches what openssl's binder verification expects.
  3. Resumption_Master_Secret: derived after CF append; check the
     transcript at that point matches what openssl computed.
**Effort:** ~2 hr debug + fix.
**Unblocks:** `psk-resumption` × 7 peers × 2 directions = up to
14 cells (modulo peer-CLI gaps).

### `rsa-pss-sign`
**Feature:** `cert-rsa-pss-sha256`.
**Status:** Driver has RSA-PSS verify (used by client to validate
peer cert).  Server-side signing path is not implemented.
**Likely effort:** ~3-4 days — RSA-PSS sign primitive in
`Tls_Core.Rsa_Pss` (mirroring HACL\* `Spec.RSAPSS.fst`), plus a
sign-side `Init_Cert_Server` variant that takes an RSA private
key.
**Unblocks:** ~14 cells (RSA-PSS server cells × 7 peers × 2
directions, minus peer-CLI gaps).
**v0.5.x scope?** Probably yes — every production CDN serves RSA
certs.  Currently flagged "verify-only" in the driver tags.

### `aes256-gcm-cell`
**Feature:** `psk-external-aes256` matrix cell.
**Status:** AES-256-GCM cipher suite is fully implemented and
proven (`Channel_Aes256`).  The matrix doesn't yet exercise it
through the orchestrator's iteration loop.
**Likely effort:** ~30 min — add to the cell-iteration tuple,
verify openssl/gnutls/mbedtls negotiate it.
**Unblocks:** 6 cells (3 peers × 2 directions).

### `zero-rtt`
**Feature:** `0-RTT early data` (RFC 8446 §2.3 + §4.2.10).
**Status:** Not implemented.
**v0.5 scope?** No.  Per CLAUDE.md §0a: 0-RTT is gated behind
opt-in flags by most peers (openssl `-early_data`, etc.); not a
production-default path.  Targeted for v0.6+ when the
production-default surface expands.
**Unblocks:** ~14 cells when implemented.

## Peer-side gaps (NOT_IMPL_3P)

These rows show up in the matrix as `NOT_IMPL_3P` — the peer
either lacks the feature or doesn't expose it via its CLI.  No
Ada-side action required.

| Feature | Peer | Reason |
|---|---|---|
| `psk-external-*` | rustls | rustls-mio CLI doesn't expose external PSK (lib does) |
| `psk-external-*` | Go | `crypto/tls` has no external-PSK API |
| `psk-external-*` | BoringSSL | `bssl -psk-hex` is RFC 9258 (Imported PSK), not RFC 8446 |
| `psk-external-*` | wolfSSL | example tools' `-s` is RFC 4279 (TLS 1.2 PSK hint) |
| `cert-rsa-pss-sha256` (server) | (any) | even if peer supports it, our server doesn't sign RSA-PSS — appears as NOT_IMPL_ADA in s2c, not NOT_IMPL_3P |

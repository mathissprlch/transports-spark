# v0.5 — TLS 1.3 (pure Ada/SPARK)

## Goal

A TLS 1.3 (RFC 8446) implementation that lives at the same proof level
as the rest of this workspace: RecordFlux for wire formats, SPARK for
the glue, gnatprove for discharge. Ambition is **platinum** —
functional correctness — on every slice we own, with the lemma
structure read off the miTLS F* sources.

## Hard constraints

1. **No C bindings.** No FFI to libmitls, HACL\*, BoringSSL, mbedTLS,
   BearSSL, OpenSSL — anything. The whole stack is Ada/SPARK.
2. **miTLS is a reference, not a dependency.** We read the F* sources
   to learn the lemmas, restate the contracts in SPARK, and discharge
   them ourselves with gnatprove. We do not vendor or link any miTLS
   artefact.
3. **Same layering as v0.4.** Wire formats live in `.rflx`; generated
   SPARK is treated as bronze; hand-written wrappers carry the
   correctness contracts.

## Architecture

```
                     ┌──────────────────────────────────────┐
                     │ Tls_Core.Transport (Ada record)      │
                     │   Connect / Send / Read / Close      │
                     └─────────────┬────────────────────────┘
                                   │
                ┌──────────────────┼──────────────────────┐
                │                  │                      │
        ┌───────────────┐  ┌────────────────┐  ┌──────────────────┐
        │ Handshake FSM │  │ Record layer   │  │ Key schedule     │
        │ (SPARK)       │  │ (SPARK)        │  │ (SPARK)          │
        │ contracts ←── │  │ contracts ←─── │  │ contracts ←──    │
        │  miTLS HS     │  │  miTLS Record  │  │  miTLS HKDF/KS   │
        └───────┬───────┘  └────────┬───────┘  └────────┬─────────┘
                │                   │                   │
                ▼                   ▼                   ▼
        ┌─────────────────────────────────────────────────────┐
        │ RecordFlux-generated wire codecs (handshake.rflx,   │
        │ record.rflx, alert.rflx, hkdf.rflx, x509-min.rflx)  │
        └─────────────────────┬───────────────────────────────┘
                              │
                ┌─────────────┴───────────────┐
                │ Crypto primitives (SPARK)   │
                │   SHA-256/384, HMAC, HKDF,  │
                │   ChaCha20-Poly1305, AES-GCM│
                │   Ed25519, X25519, P-256    │
```

Crypto primitives are the largest unknown. Options:

- **SPARKNaCl** (Roderick Chapman) — a SPARK-proven port of NaCl/TweetNaCl. Covers ChaCha20, Poly1305, X25519, Ed25519, SHA-512 at silver/gold. Doesn't cover SHA-256, AES-GCM, P-256, HMAC.
- **Hand-port a minimal slice** of HACL*'s F* code into SPARK. Specs map cleanly; HACL\* algorithm structure is straightforward.
- **Mixed:** SPARKNaCl where it covers, hand-port HMAC + SHA-256 ourselves (small).

Decision deferred until the slice that actually needs each primitive — there is no point porting AES-GCM before we have anywhere to plug it in.

## Slicing strategy

Smallest semantically-meaningful pieces, ordered by both impact and
tractability. Each slice should:

- Have a RecordFlux spec that defines the on-wire shape.
- Have a SPARK wrapper with `Pre`/`Post` that mirrors a specific
  miTLS lemma (cross-referenced in code comments).
- Discharge cleanly at gnatprove gold; aspire to platinum on the
  wrapper's correctness postcondition.

### Slice 1 — HKDF-Expand-Label (RFC 8446 §7.1) ✓

Smallest. The whole TLS 1.3 key schedule routes through it. Contract
is "the `info` argument passed to HKDF-Expand equals the §7.1 byte
concatenation."

- RecordFlux: `HkdfLabel { length: u16; label: opaque<7..255>;
  context: opaque<0..255>; }`
- SPARK wrapper: `Tls_Core.Hkdf.Expand_Label`, generic over the
  underlying HMAC-Expand primitive.
- miTLS reference: `MiTLS.HKDF.fst` `format` and `expand_label`.
  Their `assume`s on `length tls13_prefix = 6` and
  `length label_bytes >= 7` are discharged Ada lemmas.
- **gnatprove --level=4: Build_Info_Bytes 105/105 checks proved,
  zero pragma Assume.**

### Slice 2 — Derive-Secret + key schedule tree ✓

Composes Slice 1 + the transcript hash. `Tls_Core.Key_Schedule`
gives `Extract` (= HMAC-SHA-256) and `Derive_Secret` (=
Expand-Label with context = SHA-256(transcript)). Reproduces the
RFC 8448 §3 Early Secret + "derived" calculations byte-for-byte.

### Slice 3 — Record-layer AEAD wrapper ✓

Sequence-number → nonce derivation, AEAD seal/open with
`additional_data = TLSCiphertext header`. Ports miTLS
`MiTLS.Record` invariants on monotonic seq + fixed IV.
- **gnatprove --level=4: Lemma_Nonce_Injective +
  Lemma_Bump_Fresh_Nonce + Stream operations: 34/34 checks
  proved, zero pragma Assume.** The no-nonce-reuse property is
  proven from a Lemma_Roundtrip on u64 byte decomposition that
  gnatprove discharges automatically.
- Generic `Aead` package instantiates against a working
  ChaCha20-Poly1305 primitive (Slice 7); scenario 12 confirms
  two seals of identical plaintext produce distinct ciphertexts
  (the no-reuse property surfaces operationally).

### Slice 4 — Handshake message envelope (RecordFlux) ✓

`record_layer.rflx` (Content_Type + Plaintext) and
`handshake_layer.rflx` (Handshake_Type + Envelope). Thin Ada
wrapper `Tls_Core.Records` over the generated serializer; scenario
13 round-trips a TLS 1.3 record byte-for-byte against the
RFC 8446 §5.1 layout.

### Slice 5 — ClientHello / ServerHello + extension framing  *(deferred)*

Extension parsing in RecordFlux is non-trivial because of variant
content-by-tag. The PSK-only handshake path doesn't strictly need
the wire-level CH/SH because both peers know the bytes by
construction in a loopback test (and a state machine can hand them
to the transcript-hash accumulator as opaque `Handshake.body`).
For PSK-only end-to-end demo we don't need this; it lands when we
add a real-peer interop test.

### Slice 6 — Handshake state machine ✓

The big one. Mirrors miTLS `MiTLS.Handshake.fst`. Landed:
- `Tls_Core.Transcript` — incremental SHA-256 with non-destructive
  Snapshot.
- `Tls_Core.Finished` — RFC 8446 §4.4.4 verify_data computation.
- `Tls_Core.Handshake.Derive_Psk_Secrets` — full PSK_KE
  key-schedule tree from PSK + recorded transcript, producing the
  four traffic secrets.
- `Tls_Core.Handshake_Driver` — wire-level Client/Server state
  machine, two modes:
    * PSK_KE (Init)      — RFC 8446 §7.1 mode 1
    * ECDHE  (Init_Ecdhe) — RFC 8446 §7.1 mode 3 (X25519 key share
                            embedded in CH/SH body, peer pub
                            extracted on receipt, Scalar_Mult to
                            shared secret, Derive_Ecdhe_Secrets)
  Each peer's Step is a single byte buffer in / byte buffer out.
  scenario_16 confirms PSK_KE loopback reaches Done with matching
  secrets; scenario_20 confirms the same for ECDHE.

Certificate auth (Ed25519 + minimal X.509) lands later — both
PSK and PSK-less ECDHE-anon profiles are wire-level complete.

### Slice 7 — Crypto primitives ✓

- `Tls_Core.Sha256` — FIPS 180-4 §6.2 (Init/Update/Finalize/Hash).
  Matches FIPS 180-4 §B.0/§B.1/§B.2 plus streaming-split test.
- `Tls_Core.Hmac_Sha256` — RFC 2104 / FIPS 198-1, matches
  RFC 4231 §4.2 case 1.
- `Tls_Core.Hkdf_Sha256` — RFC 5869 §2.3, matches RFC 5869 §A.1
  (42-byte OKM exact).
- `Tls_Core.Chacha20` — RFC 8439 §2.3 + §2.4, matches §2.3.2
  block + §2.4.2 encrypt + decrypt round-trip.
- `Tls_Core.Poly1305` — RFC 8439 §2.5, matches §2.5.2.
- `Tls_Core.Aead_Chacha20_Poly1305` — RFC 8439 §2.8, matches
  §2.8.2 (ciphertext + tag byte-exact, Open round-trips,
  tampered ciphertext rejected via constant-time tag compare).

- `Tls_Core.X25519` — Curve25519 scalar multiplication (RFC 7748).
  16-limb, 16-bit-per-limb field representation over GF(2^255-19);
  Montgomery ladder with constant-time CSWAP; matches RFC 7748
  §5.2 test vectors byte-exact and reproduces the §6.1
  Diffie-Hellman shared-secret value (Alice & Bob both derive
  the same shared bytes from each other's public key).

Ed25519 + minimal X.509 parser deferred — needed for server
certificate paths in non-PSK handshakes; X25519 alone unblocks
ECDHE-PSK and pure-ECDHE-anon profiles.

### Slice 8 — Encrypted Channel ✓

`Tls_Core.Traffic_Keys` derives (key, write_iv) per RFC 8446 §7.3
from any traffic secret. `Tls_Core.Channel` is a connected
record-layer endpoint: Send wraps a plaintext as a TLSCiphertext
envelope (0x17 0x03 0x03 || u16 len || AEAD(pt) || tag) and bumps
the per-record sequence; Receive parses and decrypts. scenario_17
proves it round-trips three records with the derived
client_application_traffic_secret and rejects a tampered record
in constant time.

Wrapping it over a real Transport (`Http2_Core.Transport`,
`Mqtt_Core.Transport`) is one small adapter layer per backend —
deferred until we add ECDHE so we can talk to non-PSK peers.

## Lemma-porting workflow

For each slice:

1. Identify the corresponding F* file(s) in
   `https://github.com/project-everest/mitls-fstar`.
2. In the Ada wrapper, comment the source path of every borrowed
   invariant.
3. Translate the F* `requires` to SPARK `Pre`, `ensures` to `Post`.
4. Where miTLS uses `assume val ... = ...`, restate the assumption
   as an Ada `pragma Assume` with a comment naming the F* axiom and
   leave a TODO to discharge. (Often these are length facts about
   string literals, easy in Ada.)
5. Where miTLS proves a non-trivial lemma in F*, port the proof
   structure to a SPARK ghost procedure with a matching shape; let
   gnatprove discharge.

This is exactly the wrapper-with-lemmas pattern we noted at v0.4
close-out, applied to a much larger surface.

## Out of scope for v0.5

- Renegotiation (TLS 1.3 doesn't have it; nothing to do).
- Early data / 0-RTT (delayed; replay-protection state is its own
  proof effort).
- Client authentication via certificates (PSK + server-only certs
  is the v0.5 floor).
- Session tickets beyond static PSKs.
- Post-quantum hybrid groups.

## Risks

1. **Crypto-primitive proof effort.** Hand-porting AES-GCM with
   functional correctness proofs is a serious undertaking. If
   SPARKNaCl + a hand-rolled HMAC-SHA-256 covers the cipher suite
   we want (TLS_CHACHA20_POLY1305_SHA256 and
   TLS_AES_128_GCM_SHA256), we focus there. AES-GCM may simply
   stay at silver with a documented trust boundary.
2. **RecordFlux + variants.** TLS extensions are the largest
   variant-content schema we'd express in RecordFlux. May hit
   tooling gaps; mitigation is to model only the subset we
   actually negotiate.
3. **Handshake state machine size.** miTLS's handshake is hundreds
   of lemmas. Slice 6 may need to be split further.

## Verification (how we'll know it works)

- Each slice's wrapper proves at gnatprove level=4 with no `null`
  fallbacks on the contracts ported from miTLS.
- A pure-Ada-Ada loopback handshake completes (client + server
  both ours, on the in-process Transport).
- An interop run: grpcurl-over-TLS against our greeter on
  loopback, with our X25519 key share and ChaCha20-Poly1305.
- An interop run: mosquitto_pub --cafile against our broker.

## Phase 4+ — full external interop (in progress)

Acceptance criterion (the research thesis): a platinum-proven TLS 1.3
stack must complete a full handshake + application-data exchange,
in **both directions**, against **at least two unmodified reference
implementations** (`openssl s_client/s_server`, grpcurl, optionally
a browser), on **all mainstream RFC 8446 negotiation surfaces**,
and the only changes ever required during integration must be
either (a) a primitive-axiom update or (b) a spec-translation bug
fix. If the platinum spec is correct and the translation is correct,
no interop fixes should be necessary.

  above that (compositions, key derivation, AEAD construction,

  these axioms by gnatprove. Same architecture miTLS uses against
  HACL\*'s F\* primitive specifications.
- Pure Ada/SPARK throughout. No FFI, no C bindings, miTLS read
  only. Every wire format is generated from a RecordFlux spec;
  every cryptographic primitive is the FIPS / RFC pseudocode by
  inspection.

What's working as a v0.5 product right now: a Client and a Server
in-process can run a TLS 1.3 handshake to Done in either PSK_KE or
ECDHE mode through `Tls_Core.Handshake_Driver`. The ECDHE path
exchanges X25519 public keys in the Hello bodies, both peers
derive the shared secret via Curve25519 scalar multiplication, and
Derive_Ecdhe_Secrets composes that into the four traffic secrets.
Either mode opens a record-layer Channel that exchanges
authenticated, sequence-numbered application records under
ChaCha20-Poly1305.

All cryptographic primitives needed for non-PSK interop are now in
the crate at byte-exact RFC vector match:
   - SHA-256 / SHA-512 (FIPS 180-4 §B / §C).
   - HMAC-SHA-256 (RFC 4231 §4.2).
   - HKDF-Extract / Expand / Expand-Label (RFC 5869 §A.1, RFC 8446 §7.1).
   - ChaCha20-Poly1305 AEAD (RFC 8439 §2.8.2).
   - X25519 (RFC 7748 §5.2 + §6.1 DH).
   - Ed25519 verify (RFC 8032 §7.1).
   - Minimal X.509 Ed25519 self-signed cert parser.
   - Real RFC 8446 §4 ClientHello / ServerHello wire-format
     codec with the extension framing we negotiate
     (supported_versions, key_share, supported_groups,
     signature_algorithms).

Phase-1 implementation is functionally complete. What remains is
wiring CertificateVerify + Certificate handshake messages into
the driver between ServerHello and Finished (mechanical now that
Ed25519 + X.509 are in), a transport adapter, then the platinum
push (axiomatize SHA-256/512, ChaCha20, Poly1305, X25519, Ed25519
ghost-spec equivalences and let composition contracts discharge
above them) and finally real-peer interop sanity tests.

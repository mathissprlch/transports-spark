# v0.5 — TLS 1.3 (pure Ada/SPARK)

## Goal

A TLS 1.3 (RFC 8446) implementation that lives at the same proof level
as the rest of this workspace: RecordFlux for wire formats, SPARK for
the glue, gnatprove for discharge. Ambition is **platinum** —
functional correctness — on every slice we own, with the lemma
structure read off the miTLS F* sources.

## Hard constraints

1. **No C bindings.** No FFI to libmitls, HACL\*, BoringSSL, mbedTLS,
   BearSSL, OpenSSL — anything. The whole stack is Ada/SPARK.
2. **miTLS is a reference, not a dependency.** We read the F* sources
   to learn the lemmas, restate the contracts in SPARK, and discharge
   them ourselves with gnatprove. We do not vendor or link any miTLS
   artefact.
3. **Same layering as v0.4.** Wire formats live in `.rflx`; generated
   SPARK is treated as bronze; hand-written wrappers carry the
   correctness contracts.

## Architecture

```
                     ┌──────────────────────────────────────┐
                     │ Tls_Core.Transport (Ada record)      │
                     │   Connect / Send / Read / Close      │
                     └─────────────┬────────────────────────┘
                                   │
                ┌──────────────────┼──────────────────────┐
                │                  │                      │
        ┌───────────────┐  ┌────────────────┐  ┌──────────────────┐
        │ Handshake FSM │  │ Record layer   │  │ Key schedule     │
        │ (SPARK)       │  │ (SPARK)        │  │ (SPARK)          │
        │ contracts ←── │  │ contracts ←─── │  │ contracts ←──    │
        │  miTLS HS     │  │  miTLS Record  │  │  miTLS HKDF/KS   │
        └───────┬───────┘  └────────┬───────┘  └────────┬─────────┘
                │                   │                   │
                ▼                   ▼                   ▼
        ┌─────────────────────────────────────────────────────┐
        │ RecordFlux-generated wire codecs (handshake.rflx,   │
        │ record.rflx, alert.rflx, hkdf.rflx, x509-min.rflx)  │
        └─────────────────────┬───────────────────────────────┘
                              │
                ┌─────────────┴───────────────┐
                │ Crypto primitives (SPARK)   │
                │   SHA-256/384, HMAC, HKDF,  │
                │   ChaCha20-Poly1305, AES-GCM│
                │   Ed25519, X25519, P-256    │
                └─────────────────────────────┘
```

Crypto primitives are the largest unknown. Options:

- **SPARKNaCl** (Roderick Chapman) — a SPARK-proven port of NaCl/TweetNaCl. Covers ChaCha20, Poly1305, X25519, Ed25519, SHA-512 at silver/gold. Doesn't cover SHA-256, AES-GCM, P-256, HMAC.
- **Hand-port a minimal slice** of HACL*'s F* code into SPARK. Specs map cleanly; HACL\* algorithm structure is straightforward.
- **Mixed:** SPARKNaCl where it covers, hand-port HMAC + SHA-256 ourselves (small).

Decision deferred until the slice that actually needs each primitive — there is no point porting AES-GCM before we have anywhere to plug it in.

## Slicing strategy

Smallest semantically-meaningful pieces, ordered by both impact and
tractability. Each slice should:

- Have a RecordFlux spec that defines the on-wire shape.
- Have a SPARK wrapper with `Pre`/`Post` that mirrors a specific
  miTLS lemma (cross-referenced in code comments).
- Discharge cleanly at gnatprove gold; aspire to platinum on the
  wrapper's correctness postcondition.

### Slice 1 — HKDF-Expand-Label (RFC 8446 §7.1)  *(this commit)*

Smallest. The whole TLS 1.3 key schedule routes through it. Contract
is "the `info` argument passed to HKDF-Expand equals the §7.1 byte
concatenation."

- RecordFlux: `HkdfLabel { length: u16; label: opaque<7..255>;
  context: opaque<0..255>; }`
- SPARK wrapper: `Tls_Core.Hkdf.Expand_Label`, generic over the
  underlying HMAC-Expand primitive.
- miTLS reference: `MiTLS.HKDF.fst` `format` and `expand_label`.
  Their `assume`s on `length tls13_prefix = 6` and
  `length label_bytes >= 7` become discharged Ada lemmas.

### Slice 2 — Derive-Secret + key schedule tree

Composes Slice 1 and the transcript hash. Ports miTLS
`MiTLS.HKDF.derive_secret` and the `KS.fst` finite tree.

### Slice 3 — Record-layer AEAD wrapper

Sequence-number → nonce derivation, AEAD seal/open with
`additional_data = TLSCiphertext header`. Ports miTLS
`MiTLS.Record.Stream`/`MiTLS.Record.Encryption` invariants. The
no-nonce-reuse property here is the one the user can quote.

### Slice 4 — Handshake message envelope (RecordFlux)

Just the `Handshake { msg_type; length; body }` framing and the
`TLSPlaintext`/`TLSInnerPlaintext` records.

### Slice 5 — ClientHello / ServerHello + extension framing

Extension parsing in RecordFlux is non-trivial because of variant
content-by-tag. Spec-side burden but no proof obligations beyond
parse-roundtrip.

### Slice 6 — Handshake state machine

The big one. Mirrors miTLS `MiTLS.Handshake.fst` invariants on legal
state transitions, transcript-hash growth, and key changeover. v0.5
target is the unilateral-PSK-resumption case (smallest valid handshake)
plus full 1-RTT with X25519 key share and Ed25519 certs.

### Slice 7 — Crypto primitives (whichever the above demand)

SHA-256 + HMAC are required from Slice 1. ChaCha20-Poly1305 from
Slice 3. Defer the rest.

### Slice 8 — Transport adapter

`Tls_Core.Transport` over `Http2_Core.Transport` and
`Mqtt_Core.Transport`, so the existing v0.4 stack can run on TLS.
Test with grpcurl-over-TLS and mosquitto-over-TLS.

## Lemma-porting workflow

For each slice:

1. Identify the corresponding F* file(s) in
   `https://github.com/project-everest/mitls-fstar`.
2. In the Ada wrapper, comment the source path of every borrowed
   invariant.
3. Translate the F* `requires` to SPARK `Pre`, `ensures` to `Post`.
4. Where miTLS uses `assume val ... = ...`, restate the assumption
   as an Ada `pragma Assume` with a comment naming the F* axiom and
   leave a TODO to discharge. (Often these are length facts about
   string literals, easy in Ada.)
5. Where miTLS proves a non-trivial lemma in F*, port the proof
   structure to a SPARK ghost procedure with a matching shape; let
   gnatprove discharge.

This is exactly the wrapper-with-lemmas pattern we noted at v0.4
close-out, applied to a much larger surface.

## Out of scope for v0.5

- Renegotiation (TLS 1.3 doesn't have it; nothing to do).
- Early data / 0-RTT (delayed; replay-protection state is its own
  proof effort).
- Client authentication via certificates (PSK + server-only certs
  is the v0.5 floor).
- Session tickets beyond static PSKs.
- Post-quantum hybrid groups.

## Risks

1. **Crypto-primitive proof effort.** Hand-porting AES-GCM with
   functional correctness proofs is a serious undertaking. If
   SPARKNaCl + a hand-rolled HMAC-SHA-256 covers the cipher suite
   we want (TLS_CHACHA20_POLY1305_SHA256 and
   TLS_AES_128_GCM_SHA256), we focus there. AES-GCM may simply
   stay at silver with a documented trust boundary.
2. **RecordFlux + variants.** TLS extensions are the largest
   variant-content schema we'd express in RecordFlux. May hit
   tooling gaps; mitigation is to model only the subset we
   actually negotiate.
3. **Handshake state machine size.** miTLS's handshake is hundreds
   of lemmas. Slice 6 may need to be split further.

## Verification (how we'll know it works)

- Each slice's wrapper proves at gnatprove level=4 with no `null`
  fallbacks on the contracts ported from miTLS.
- A pure-Ada-Ada loopback handshake completes (client + server
  both ours, on the in-process Transport).
- An interop run: grpcurl-over-TLS against our greeter on
  loopback, with our X25519 key share and ChaCha20-Poly1305.
- An interop run: mosquitto_pub --cafile against our broker.

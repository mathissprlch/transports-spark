# v0.5 Tier D — external interop matrix

## Scope

This document is the **empirical bridge** for v0.5: every cell is a
real wire-protocol experiment between our Ada/SPARK TLS 1.3 stack
(`Tls_Core.Tls13_Driver`) and a production peer (currently OpenSSL
3.6.1).  CLAUDE.md §6 (honest reporting language) applies — every
row reports what the experiment actually showed, not what we wished
it would show.

The harness lives under:

- `crates/examples/src/tls_interop_{client,server}.adb` — Ada
  binaries that drive `Tls_Core.Tls13_Driver` over a real TCP
  socket via `Tls_Core.Tcp_Transport`.  Plain Ada (no `SPARK_Mode`
  aspect): they are test harnesses, not part of the proven core.
  CLAUDE.md §0d preserved: no `SPARK_Mode (Off)` outside
  `Tcp_Transport`, no `pragma Assume`, only public `Tls_Core.*`
  APIs.
- `scripts/interop/{gen_pki,openssl_server,openssl_client,
  run_matrix}.sh` — openssl-side scaffolding and the matrix runner.
- `crates/tls_core/tests/fixtures/interop/{ec,rsa}/` — checked-in
  PKI fixtures (P-256 ECDSA + RSA-PSS roots and leaf certs for
  `localhost`).

## Reproduction

```sh
# 1. Build the Ada interop binaries (one-time):
SDK=$(xcrun --show-sdk-path) SDKROOT=$SDK LIBRARY_PATH=$SDK/usr/lib \
  alr -C crates/examples build

# 2. Re-generate PKI fixtures (only if missing or expired):
./scripts/interop/gen_pki.sh

# 3. Run the full matrix (~10 s on localhost):
./scripts/interop/run_matrix.sh
```

`run_matrix.sh` produces a Markdown table on stdout and dumps
per-cell logs to `/tmp/spark-tls-interop/<timestamp>/`.  The
table block below is captured from a representative run on
2026-05-08; the runner re-emits it on every invocation.

## Identified gaps (root-cause notes)

### GAP-D-1 — PSK binder hashed over body-only, not handshake-message-formatted CH

**Severity: ★★★ blocks ALL openssl PSK cells (both directions, all
cipher suites).**

RFC 8446 §4.4.1 + §4.2.11.2 require:

> binder = HMAC(finished_key, Transcript-Hash(Truncated(ClientHello1)))

where `Truncated(ClientHello1)` is the **Handshake structure** (i.e.
type + 3-byte length + body), truncated just before the binders
list.  Both `Tls_Core.Tls13_Driver`'s client emit path
(`tls13_driver.adb:397`) and the server verify path
(`tls13_driver.adb:977`) call `Psk_Binder.Compute` with the **CH
body only** — the 4-byte handshake header is not prepended.

Ada-vs-Ada works because both sides compute the same (wrong) hash.
OpenSSL hashes the spec-compliant bytes and rejects every binder.

Suggested fix (small, internal): in `Tls_Core.Psk_Binder.Compute`,
allow the caller to pass the message-formatted bytes; or, more
mechanical, have `Tls13_Driver` build a 4-byte handshake header
prefix and concatenate before calling Compute.  Wire-tested against
`openssl s_server -psk` is the validation gate.

### GAP-D-2 — `-ciphersuites TLS_AES_256_GCM_SHA384` triggers a key_share path on openssl

When openssl is constrained to SHA-384 mode, its CH parser hits a
distinct "length mismatch" branch on the key_share extension before
even reaching the PSK-binder code.  Likely interaction: openssl is
processing extensions in a different order under SHA-384 because
the ext-binder PRF length is 48 instead of 32, and our wire
key_share's `key_exch_len = 32` collides with that path's length
expectation.  Need wire-trace investigation.  **Strictly downstream
of GAP-D-1 — fixing the binder will likely make this cell either
PASS or move to a different, smaller failure.**

### GAP-D-3 — driver server-side rejects AES_256_GCM_SHA384 by design

Documented in `Tls13_Driver.ads` package comment (the "Wall-hit"
note): the server falls into Failed when the client offers ONLY
AES_256_GCM_SHA384 because the SHA-384 key-schedule path is not
yet ported.  This is independent of GAP-D-1 — fixing the binder
won't unblock this cell.  Tracked separately in the v0.5 plan.

### GAP-D-4 — cert-mode interop has no driver entry point

`Tls_Core.Tls13_Driver` is **PSK-only**.  Cert-mode (server-cert
authentication, RFC 8446 §4.4.2) lives in the older
`Tls_Core.Handshake_Driver`, which does **not** emit RFC-8446-§5
TLSPlaintext / TLSCiphertext records — its records are simplified
in shape and are not consumable by openssl.

The v0.5-tls-plan.md tracks Phase 14 ("Real Cert + CertVerify with
full DER chain") for this work.  Until that lands, cert-mode
cells in the matrix are deferred.  **Blocking item for v0.5
release-readiness against the original v0.5 scope; the harness
binaries are ready to consume an `Init_Cert_*` API the moment
it appears.**

### GAP-D-5 — HRR cell harness not yet plumbed

`Tls13_Driver` does support HRR (`Init_Psk_Server_With_Hrr` +
`Init_Psk_Client_Hrr_Aware`), but `tls_interop_client` always
calls `Init_Psk_Client` (non-HRR-aware) and the matrix runner
hasn't yet been generalized to a tri-state CLI flag.  Mechanical
follow-up: add `--hrr-aware` to the client and a one-shot HRR
cell to the matrix.

### GAP-D-6, GAP-D-7 — SNI / ALPN extensions not advertised

`Tls_Core.Hello.Encode_Client_Hello_Psk` does not emit `server_name`
(SNI, type 0x0000) or `application_layer_protocol_negotiation`
(ALPN, type 0x0010) extensions.  Adding them is mechanical wire-
format work in `Tls_Core.Hello`; matrix cells are stubbed to
remind us they remain in v0.5 scope per CLAUDE.md §0a.

## Phase D2 deferrals

The following peers are explicitly out of scope for this dispatch
and will be addressed in subsequent runs:

| Peer | Reason for deferral |
|---|---|
| rustls (CLI) | Separate dispatch — needs rustls-cli or a small Rust harness binary. |
| Go crypto/tls | Separate dispatch — needs a small Go program. |
| mbedTLS | Likely Docker, separate dispatch. |
| BoringSSL | Lower priority; chrome-internal. |
| GnuTLS | Lower priority; decreasing share. |

The expectation per CLAUDE.md §0a is that **once the GAP-D-1
binder fix lands**, openssl PSK interop will most likely work, and
the same fix will be the unlocking gate for rustls / Go / mbedTLS
PSK cells (they all follow RFC 8446 §4.2.11.2 to the byte).

## Tier D readiness assessment for v0.5 release

- **Harness scaffolding**: complete — Ada interop binaries +
  openssl wrappers + matrix runner all land in this dispatch.
- **Empirical bridge**: NOT yet established.  Zero openssl cells
  pass.
- **Identified gaps**: 7 named, with root-cause analysis.  The
  smallest one (GAP-D-1) is a one-function-rewrite that would
  most likely unlock 4–6 of the openssl cells immediately.

This matches CLAUDE.md §6's distinction between "harness landed"
and "interop tested."  Tier D is **scaffolded** but not yet
**delivered**; the matrix doc names every blocker so the next
session has a concrete ordered checklist:

  1. GAP-D-1 (binder hash) — highest leverage, smallest fix.
  2. GAP-D-3 (SHA-384 key schedule) — unlocks the AES-256 row.
  3. GAP-D-4 (cert-mode driver entry points) — unlocks 3 cells +
     the rustls / Go interop matrices that don't speak external PSK.
  4. GAP-D-5/6/7 (HRR + SNI + ALPN harness/wire wiring) — small
     each, complete the "production-default" surface.
